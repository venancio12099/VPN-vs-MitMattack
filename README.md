# Mitigating Man-In-The-Middle attacks using a VPN.
In many real-world environments, users connect to corporate services from untrusted networks such as cafés, airports or client offices. 
If the company still exposes internal services over HTTP or other unencrypted protocols, an attacker on the same network could intercept or manipulate the traffic using a Man-in-the-Middle (MITM) attack.

This lab recreates that scenario in a controlled environment using virtual machines. 
One machine acts as the victim (Windows 10), and the other acts as the attacker/sniffer (Kali Linux). 
The goal is to demonstrate how a VPN can prevent MITM attacks by encrypting all traffic before it reaches the local network.

# Objectives
- Understand how Arp Spoofing attacks work.
- Understand how to prevent MITM attacks
- Review how a VPN encrypts traffic and prevents packet inspection.

# Tools Used In This Lab
- VMware Workstation (Virtualization envoirement)
- Bettercap (ARP spoofing and network manipulation)
- Wireshark (Packet capture and protocol analysis)
- ProtonVPN (VPN client for encrypted tunneling)

# Lab Set Up
For this lab, I used two virtual machines connected within the same virtual network.
Both machines share the same Layer 2 broadcast domain, which is required for ARP spoofing.

- Windows 10 (victim) with IP: 192.168.205.129
- Kali Linux (attacker/sniffer) with IP: 192.168.205.128

Since both hosts can reach each other, the attacker can poison the ARP tables of the victim and the gateway.
This allows the attacker to intercept and forward traffic.

# ARP Spoofing Attack

Using Bettercap, the attacker send forged ARP replies to both the victim and the gateway.
Consecuences:
- The victim believes the attacker’s MAC address belongs to the gateway’s IP.
- The gateway believes the attacker’s MAC address belongs to the victim’s IP.
- Creation of a bidirectional MITM channel, allowing the attacker to capture all traffic flowing between both endpoints.
  
<img width="1912" height="1015" alt="arp_spoof on_" src="https://github.com/user-attachments/assets/1703287d-5485-47aa-8b1f-55af829b7ca0" />

Once the spoofing is active, Wireshark capture all packets generated by the victim.
Since the victim is browsing over HTTP, the attacker could capture and see things in plain text such as credentials, cookies, downloaded files...
This demonstrates how dangerous ARP spoofing can be when combined with unencrypted protocols.

# Mitigation Using a VPN

To mittigate this attack, I made use of ProtonVPN on the victim machine.

<img width="697" height="495" alt="vpn-activation" src="https://github.com/user-attachments/assets/359792d6-6a64-401e-b390-1c737a40a0af" />

After enabling the VPN, the victim carries on using the HTTP service, however, Wireshark no longer displays HTTP traffic.
Even though the attacker is successfully performing ARP spoofing, the VPN prevents any interception as the traffic is encrypted end‑to‑end.
Resulting in no HTTP traffic, credentials, or readable data.

<img width="1914" height="1015" alt="no_package_capture" src="https://github.com/user-attachments/assets/a590d5b2-86b2-40eb-8fff-57a394d9eb58" />

The packets appear now as ISAKMP, this is because ProtonVPN is likely using IKEv2 (a common VPN protocol) or another service that defaults to UDP port 500.
Wireshark often marks some of them as “Malformed” because it find random encrypted bits and realizes they don't follow the ISAKMP rules.

<img width="1918" height="1015" alt="spoof_mitigated" src="https://github.com/user-attachments/assets/fc1f14e0-a593-4a65-b3dd-c064d71500e2" />

Looking deeper into the ISAKMP packets confirm that the contents are encrypted and unreadable.

<img width="1919" height="986" alt="encrypted_package" src="https://github.com/user-attachments/assets/69d1a409-85de-4291-9564-3e3b0212dcd0" />

# Conclusion

This lab demonstrates how ARP spoofing can be used to perform MITM attacks on unencrypted HTTP traffic, and how a VPN effectively neutralizes the attack by encrypting all traffic before it reaches the local network.
